## 1. `docs/bug-reports.md` — v2 формат (8 полей)

### Каноническая строка
```
- <id> | <date> | <description> | <severity> | <category> | <status> | <fix_attempts> | <comment>
```

### Поля
| # | Поле | Тип | Обяз. | Значения |
|---|------|-----|:---:|----------|
| 1 | `id` | String `B-NNN` или `VB-NNN` | ✓ | `B-042` — назначает приложение (счётчик), LLM не трогает |
| 2 | `date` | ISO `YYYY-MM-DD` | ✓ | `2026-04-19` — назначает приложение на создании, immutable |
| 3 | `description` | String с escape `\|` и `\n` | ✓ | `App crashes on login` — создаёт user, LLM не трогает |
| 4 | `severity` | Enum | ✓ | `critical` / `major` / `medium` / `minor` / `trivial` |
| 5 | `category` | String (free-form) | ✓ | `ui_ux` / `ux_flow` / `logic` / `auth` / `database` / `performance` / `security` / `integration` / `other` — зависит от роли репо (см. BugItem.svelte `categoriesByRole`) |
| 6 | `status` | Enum | ✓ | `created` → `in-progress` → `testing` → `confirmed` (delete row) / `rejected` (+attempt) |
| 7 | `fix_attempts` | Integer ≥ 0 | ✓ | `0`, `3` — инкрементируется приложением при `rejected` transition |
| 8 | `comment` | String (может быть empty) | — | Пусто до первого fix-attempt'а; LLM вписывает что сделано |

### Escape
- Литеральный `|` внутри description/comment → `\|`
- Литеральный newline → `\n`
- Parse: `split_pipe_respecting_escape` (negative-lookbehind на `\`), затем unescape `\n` → newline, `\|` → `|`

### LLM/AI policy
LLM меняет **только** `status` и `comment`. Всё остальное:
- `id`, `date` — **immutable** (назначает app на создании)
- `description`, `severity`, `category` — **user-owned** (устанавливает user)
- `fix_attempts` — **app-managed** (инкрементируется при rejected)

### Workflow статусов
```
created → in-progress → testing → confirmed (delete from file)
                                → rejected → in-progress (attempts +1)
```

### Что принимает parser
- Префикс `- [ ] ` ИЛИ `- ` с id начинающимся на `B-`/`VB-`
- 8 полей = v2 (нормальный путь)
- 10 полей = legacy v1 (`screen + description + reproduction` склеиваются в один `description`). Будет удалено в v0.14.0 после миграции всех файлов пользователя.
- Другое число полей → warning, строка пропущена

### Что пишет generator
Всегда v2 (8 полей), с заголовком `# Bug reports` + workflow + escape-note + `## Open bugs`. Подтверждённые (`confirmed`) вызывающим кодом исключаются.

## 2. `docs/todo.md` — 5 полей

### Каноническая строка
```
- [ ] <id> | <description> | <effort> | <priority> | <status>
```

### Поля
| # | Поле | Тип | Обяз. | Значения |
|---|------|-----|:---:|----------|
| 1 | `id` | String, обычно `T-NNN` или `F-NNN` | ✓ | `T-042`, `F-022` |
| 2 | `description` | String с escape `\|` и `\n` | ✓ | `Add Go deploy template` |
| 3 | `effort` | Free-form tag | ✓ | `S` / `M` / `L` (размер) **или** часы `2` |
| 4 | `priority` | Enum | ✓ | `must` / `should` / `could` |
| 5 | `status` | Enum | ✓ | `open` / `in-progress` |

### Что принимает parser (tolerant)
- Префикс `- [ ] ` ИЛИ `- ` с id начинающимся на `T`/`F`/`B`/`v`/цифру
- 3 поля: `id | description | status` (effort/priority пустые)
- 4 поля: `id | description | effort | status` (priority пустой)
- 5 полей: канонично
- 6+ полей: `id | middle₁ | middle₂ ... | effort | priority | status` — middle_n склеивается в description через ` | `
- <3 полей → warning

### Что пишет generator
**Генератора нет.** `docs/todo.md` — user-authored. Приложение только читает. Это значит: user следит за форматом сам.

### LLM/AI policy
Currently user-owned. Direct LLM edits out of scope (может предлагать, но не правит).

### ⚠️ Расхождение с глобальным `~/.claude/CLAUDE.md`
Там указано:
```
ID | Описание | Трудоёмкость (в часах) | Критичность (Очень высокая, высокая, средняя, низкая) | Статус (открыта/выполняется)
```
vs. текущая реальность:
- `effort` — S/M/L или часы (свободно), не жёстко "часы"
- `priority` — `must/should/could` (English 3-tier), не "Очень высокая/высокая/средняя/низкая" (Russian 4-tier)
- `status` — `open/in-progress` (English), не "открыта/выполняется" (Russian)

**Решение нужно:** какой источник истины? Если глобальный CLAUDE.md — переводим приложение под русские enum-ы. Если приложение — обновляем CLAUDE.md. Моё предложение — держать English enums (они короче, удобнее парсить, совпадают с bug-reports `severity`), и обновить CLAUDE.md.

## 3. `docs/done.md` — 4 поля

### Каноническая строка
```
- [x] <id> | <description> | <date> | <commit>
```

### Поля
| # | Поле | Тип | Обяз. | Пример |
|---|------|-----|:---:|--------|
| 1 | `id` | String | ✓ | `T-042`, `F-022`, `v0.13.0` |
| 2 | `description` | String с escape | ✓ | `Go deploy template added` |
| 3 | `date` | `YYYY-MM-DD` или `DD.MM.YYYY` | ✓ | `2026-04-19` |
| 4 | `commit` | Short-SHA / tag / empty | — | `abc1234`, `v0.13.0`, `` |

### Что принимает parser (date-anchor heuristic)
- Префикс `- [x] ` / `- [X] ` / `- ` (если id на `T`/`F`/`B`/`v`/цифру)
- Ищется поле, похожее на дату (`YYYY-MM-DD`, `DD.MM.YYYY`, `DD/MM/YYYY`)
- Если date найден на индексе `d ≥ 1`: `id = parts[0]`, `description = parts[1..d] join(" | ")`, `date = parts[d]`, `commit = parts[d+1..] join(" | ")`
- Если date не найден: fallback на позиционный parse для 3-4 полей
- <2 полей → warning

### Что пишет generator
**Генератора нет.** `docs/done.md` — user-authored.

### ⚠️ Известная неаккуратность parser'а
Для строк БЕЗ id (`- [x] Scaffold Tauri v2 | 29.03.2026 | abc1234`) date-anchor видит дату на индексе 1 → берёт parts[0]=Scaffold... как id, description=empty. В UI выглядит как "id = описание задачи". Это не hang/panic, но некрасиво. **Решение:** либо обязать всегда писать id (и хуже читающиеся done.md старых проектов), либо сделать умнее: если parts[0] не похож на id-формат (нет префикса `T-`/`F-`/`v`/B-`), считать его description и оставлять id пустым. **Твоё мнение?**

### LLM/AI policy
Append-only. Задачи уезжают сюда при `confirmed`. Прямые правки редкие.

### ⚠️ Расхождение с глобальным `~/.claude/CLAUDE.md`
Там:
```
ID | Описание | Дата завершения | Коммит/PR
```
Совпадает ✓. Примеры в CLAUDE.md используют `YYYY-MM-DD`, это же использует приложение. OK.
